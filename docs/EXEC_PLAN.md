## 阶段 16：主播趋势汇总执行口径

### 1. 统计范围与身份隔离

- 固定周期只支持 `1/3/5/7/15/30` 天。
- 当前周期为截至 `end_date` 的完整自然日区间；基准周期为其前一日结束的紧邻等长自然日区间，不跨房间或主播借样本。
- 自动任务按 T+1 次日 08:00 截止选择最近可比较业务日：08:00 前最多使用前天，08:00 起可使用昨天；显式传入仍未结束的日期时必须判为数据不完整，而不是生成正式榜单。
- 聚合键为 `room_id + actual_anchor_canonical`。组合主播以标准组合展示名作为一个主播系列，不拆成成员后跨组合合并；房间相同但主播不同、主播相同但房间不同均隔离。

### 2. 指标、目标与样本门槛

- 当前/基准 ROI 均为 `SUM(period_overall_amount) / SUM(period_spend)`，禁止平均小时 ROI。
- 当前/基准订单成本均为 `SUM(period_spend) / SUM(period_overall_orders)`；分母为 0 或缺失时返回空值。
- ROI 目标按有效日期内的房间 ID、精确房间名、品类顺序匹配；无目标时目标差与达标状态为空，不虚构目标。
- 正式判断同时检查当前与基准周期完整率、有效小时数；当前周期还检查最低消耗和最低订单数。
- 当前或基准周期不完整、完整率不足、有效小时不足、当前消耗/订单不足、基准 ROI 或基准消耗为 0/空时，归入“样本不足/不可比”，不计算虚假的增长率。

### 3. 分类、排序与持久化

- 先匹配下跌/红色主状态，再匹配上涨状态；第一个命中的状态作为主状态，其余命中原因保留在原因列表。
- 计算结果可为 `rise`、`fall`、`insufficient` 或 `neutral`。上涨、下跌、样本不足分别形成全量汇总事件；无明显变化 `neutral` 当前不创建汇总事件。
- 上涨按 ROI 增长率降序，下跌按 ROI 增长率升序，样本不足按直播间和主播排序。
- 飞书卡片只展示规则 TopN，且硬上限为 10；`anchor_trend_items` 必须保存该类事件的全量主播结果。单次规则运行最多发送上涨和下跌两张卡，样本不足只留系统记录。

### 4. 调度、去重和重发

- 默认 3 天规则为每日 09:30，默认 7 天规则为周一 09:40；业务时区为 Asia/Shanghai。Beat 每 5 分钟检查一次，命中 15 分钟窗口，事件唯一键阻止窗口内重复创建。
- 普通汇总去重键维度为：规则 ID、周期天数、当前周期开始/结束、目标群、通知类型；值为上述字段拼接后的 SHA-256。
- `force_resend` 只允许管理员使用，必须填写原因；系统创建新的 `anchor_trend_events`，记录 `manual_resend=true`、`source_event_id`、`resend_reason` 和操作人，并复制原事件全量明细。不得把原 `sent` 事件直接改回待发送。
- 只有真实飞书 transport 返回且 `mocked=false` 才可记为 `sent`；Mock 为 `skipped`，异常为 `failed`。

## 阶段 18：五角色 RBAC 与直播间 Data Scope

### 1. 增量模型与兼容策略

- 保留 `users`、`roles`、`user_room_permissions` 与现有 OAuth 会话，新增正式 `room_resources`、`permissions`、`role_permissions`、`user_roles`、`role_room_scopes`、`feishu_groups`、`feishu_group_room_scopes`、`permission_audit_logs`。
- `UserRoomPermission` 继续承担个人特殊房间授权；用户存在个人授权时覆盖角色房间范围，否则使用角色范围；`developer` 为 ALL。
- 旧 `admin/operator/viewer` 数据分别兼容到 `admin/live_manager/viewer`；正式管理员不再被映射为开发者，避免历史兼容逻辑造成越级。

### 2. 后端安全边界

- 每个业务 API 先验证权限点，再从服务端 AccessScope 自动注入允许房间；客户端不传房间也只能查询允许范围。
- 客户端显式请求未授权 `room_id` 一律返回 403，而不是静默扩大、先查全量后过滤或依赖前端隐藏。
- 总览、小时趋势、小时比较、主播/场控/搭配、详情、透视、比较、导出、普通预警、主播趋势均使用同一房间范围。
- 飞书群必须绑定允许直播间；事件或汇总包含群范围外直播间时发送前失败关闭，禁止跨房间泄露。
- 用户、角色和系统配置由 `developer/admin` 管理；`operations_lead` 只访问数据源、同步、ROI/预警规则和审计；修改权限与敏感配置写入审计日志，值中不得出现密码或 Token。

### 3. 前端与验收

- `/auth/me` 返回权限点、角色集合、可见直播间与数据范围标签；菜单、同步、导出、管理按钮按权限显示。
- `/admin/*` 使用路由级权限保护；无权直达时显示中文 403，不发起管理 API 请求。
- 权限管理页展示用户、角色等级、直播间资源和角色 × 权限矩阵，并按服务端返回值禁用当前、同级和更高等级账号操作。
- 建立 `developer_test`、`live_manager_test`、`water_pm_test`、`primer_pm_test`、`powder_pm_test` 五类测试身份；后端真实数据范围测试与前端截图均使用隔离测试运行时，不向真实飞书群发送消息。

## 已知实现边界

- `hourly_comparison_rules.schedule_timezone` 已持久化，但当前调度计算读取全局 `settings.timezone`，尚未实现每条规则独立时区；运行配置必须保持 `Asia/Shanghai`。
- `evaluation_delay_minutes`、`cooldown_minutes`、`send_empty_summary` 已存储，但主播趋势汇总当前依赖完整自然日、事件唯一键和非空结果，不应将这些字段描述为已生效的独立调度语义。
- 数据质量规则的 `technical_push_enabled/technical_chat_id` 已建模，但当前代码只执行系统记录并阻断业务群推送，尚未实现技术群发送。
- 主播趋势严格响应 schema 已完成；列表与重算使用不同项目模型，避免用大量可选字段掩盖结构差异。`provider` 和卡片 payload 因外部协议保持为受控的不透明 JSON 字段。
- ROI 目标候选已强制限定 `metric_code=period_overall_roi`；同房间/品类的非 ROI 目标不会参与趋势目标匹配，并有回归测试。
- 受限用户的事件 `room_scope`、`anchor_count` 会按可见明细重建；混合房间事件不向受限用户返回原始 `message_snapshot`，避免卡片快照携带越权主播。

## 阶段 24：网页账号密码登录与分级访问

### 1. 认证与安全边界

- 保留飞书 OAuth 供管理员授权真实数据源和现有账号继续使用；新增独立 `POST /auth/password/login`，同事只需在驾驶舱网页输入管理员创建的登录名和密码。
- 密码只保存为随机盐的高成本单向哈希，API、审计日志和管理列表不得返回密码或哈希；错误账号与错误密码统一返回相同提示，避免账号枚举。
- 登录成功继续签发原有 8 小时 `HttpOnly`、`Secure`、`SameSite=Lax` 签名会话及独立 CSRF Cookie；角色、权限点和直播间范围仍由服务端 `AccessScope` 执行。
- 登录失败按来源与账号组合限流；生产 Redis 可用时共享计数，Redis 不可用时退化到进程内计数，不因限流依赖故障绕过密码验证。

### 2. 管理与前端

- 用户创建必须填写登录名、姓名、初始密码、角色及数据范围，邮箱改为可选；已有飞书用户可由开发者重置网页密码以启用双登录方式。
- 用户列表仅展示“网页账号已启用/未启用”和“飞书已绑定/未绑定”，提供单独的密码重置动作；每次创建、重置和网页登录成功写入不含秘密的权限审计记录。
- 未登录共享链接显示账号密码表单，同时保留“管理员使用飞书登录”入口；登录后返回原请求页面，页面数据和菜单按账号角色及直播间范围展示。
- 提供网页退出登录入口，退出后清理签名会话与 CSRF Cookie。

### 3. 验收

- 后端测试覆盖哈希不可逆与校验、正确/错误/停用账号登录、会话 `auth_mode=password`、登录限流、管理员创建及重置密码、普通账号禁止管理接口。
- 前端测试覆盖未登录表单、账号密码提交、管理员飞书入口、用户创建密码字段与网页登录状态展示。
- 每阶段更新 `docs/PROGRESS.md`，最终运行 `make.cmd check` 与 `make.cmd verify-production`；部署前备份生产数据库，公网冒烟使用临时随机密码并在验证后清除。

## 阶段 25：长期免密进入与滚动会话

### 1. 会话口径

- 网页账号和飞书账号首次成功登录后签发持久化 Cookie；默认有效期由 `SESSION_MAX_AGE_DAYS=30` 控制，允许 1–365 天配置。
- 应用启动访问 `/auth/me` 时，在原会话仍有效且账号可用的前提下滚动续期；同一浏览器关闭后再次打开链接可直接进入，不在浏览器保存密码。
- 用户主动退出时删除会话与 CSRF Cookie；账号停用、角色和直播间权限变更仍在每次服务端请求时立即生效。

### 2. 验收

- 后端测试覆盖默认 30 天 Cookie、`/auth/me` 滚动续期、可配置期限上下界和退出清理。
- 前端登录页明确提示本设备保持登录与主动退出行为；最终运行 `make.cmd check` 和 `make.cmd verify-production`。

## 阶段 27：立即同步网关超时修复

### 1. 故障口径

- 先分别核验飞书凭据、用户授权、数据源配置和公网请求时长；只有授权状态接口失败时才提示重新授权。
- 手动同步不得把完整的飞书读取、清洗和入库过程阻塞在单次公网 HTTP 请求内，避免 Netlify/Cloudflare 等代理先于后端完成而返回 504。

### 2. 实现与交互

- `POST /auth/feishu/sync` 只负责创建或复用后台任务并立即返回 HTTP 202；`GET /auth/feishu/sync/{job_id}` 提供排队、运行、完成和失败状态。
- 后台任务继续复用正式 `live_actual` 同步管线，不改变数据源、字段映射、清洗或幂等入库口径；同一进程同一时刻只执行一个手动同步任务。
- 前端收到任务后每 2 秒轮询，完成后刷新飞书状态与页面数据；失败时显示服务端返回的实际原因，不再统一误报“请检查飞书授权状态”。短暂网络断开允许连续重试 4 次，第 5 次仍失败才提示状态查询中断。

### 3. 验收

- 后端测试覆盖 HTTP 202、排队到完成、结果摘要以及重新授权失败的安全错误信息；前端测试覆盖完成轮询、失败透传和短暂断线恢复。
- 用正式公网入口执行一次真实手动同步，验证受理请求不再超时、任务最终完成、同步来源数与小时事实数可复核；测试不触发飞书群预警推送。
- 更新 `docs/PROGRESS.md` 和 `docs/TEST_REPORT.md`，运行完整 `make.cmd check` 与 `make.cmd verify-production` 后再发布。

## 阶段 28：后台账号与密码自助维护

### 1. 功能与权限边界

- 在现有“权限管理 → 用户管理”中为每个用户提供“账号密码”入口，包括尚未配置网页登录名的飞书用户。
- 开发管理员可修改网页登录名；新密码为可选项，留空时保留原密码，填写时完成二次确认并使旧密码立即失效。
- 修改凭据不得改变用户身份、飞书绑定、角色、启停状态或直播间范围；现有签名会话继续以用户 ID 识别，后续网页登录使用新登录名和新密码。

### 2. 服务端安全

- 登录名去除首尾空格、统一为不区分大小写的规范值，并在数据库事务内排除当前用户后检查唯一性。
- 密码继续使用随机盐高成本单向哈希，API、管理列表和审计日志均不返回密码或哈希；审计只记录登录名、网页登录是否启用和是否轮换凭据。
- 凭据更新接口继续执行 `permission.manage`、签名会话和 CSRF 校验；登录名冲突返回明确的 HTTP 409，不依赖前端校验保证安全。

### 3. 验收

- 后端测试覆盖同时修改登录名和密码、仅改登录名时保留哈希、重复登录名拒绝以及审计无明文；前端测试覆盖无登录名飞书用户可开通、表单二次确认和 API 路径。
- 完整运行 `make.cmd check` 与 `make.cmd verify-production`，部署前在线备份生产 SQLite，发布后验证线上入口包含“账号密码”交互且 `/ready` 正常。

## 风险与处理

- 飞书凭据和用户令牌不得出现在代码、文档、日志或测试输出中；自动测试使用可注入 Mock HTTP transport，真实联调只执行受控只读/指定群操作。
- fixture 日期跨度与排班月份不同：同步与清洗分别验证，不伪造二者之间不存在的计划关联；演示小时事实只来自真实 fixture。
- 本机无 Docker：仍生成完整 Compose、Dockerfile、健康检查和静态生产验证；最终明确运行态未在本机实测。
- Windows 无 GNU Make：提供 Makefile 与跨平台脚本，并使用仓库 `make.cmd` 执行同名目标；若未运行完整目标，必须保持为待验收。
- 本机无 Redis 且 WSL 无发行版：不能启动真实 Celery Worker/Beat。最小恢复路径是提供外部 Redis，或安装 Debian WSL 与 `redis-server` 并把 Worker/Beat 一并放入 Linux；Celery 官方不支持 Windows，`solo` 池仅可作为本地验证。

## 阶段 30：五级角色层级与防越级授权

### 1. 角色和能力

- 建立 `developer → admin → operations_lead → live_manager → project PM` 五级链；保留 `viewer` 作为旧账号只读兼容角色。
- 开发者为唯一最高等级；管理员负责用户、权限和系统；运营负责人负责全部直播间的数据源、同步、目标、规则和审计；直播主管负责经营查看和预警处置；各项目 PM 只访问所属直播间。
- 管理后台按权限点显示具体入口，运营负责人不能看到用户、系统设置、指标和班次配置。

### 2. 服务端安全

- 账号创建、角色调整、凭据修改、停用和删除统一比较操作者与目标角色等级；非开发者不能操作同级或更高等级账号。
- 角色权限配置使用最小等级上限，禁止把低等级角色改成隐藏管理员；`database.manage` 仅开发者可持有。
- 旧 `admin → developer` 兼容映射改为 `admin → admin`，种子回填会移除旧管理员被错误附加的开发者角色。
- 飞书自动开户角色限制在直播主管、项目 PM 和受限查看者。

### 3. 验收

- 后端测试覆盖角色种子、房间范围、管理员管理下级、同级/上级越权拒绝、权限上限和运营负责人分区入口。
- 前端测试覆盖五级矩阵、禁用态以及运营负责人只看到数据源、预警规则和审计。
- 更新 `docs/PERMISSIONS.md`、`docs/PROGRESS.md` 与 `docs/TEST_REPORT.md`，最终运行 `make.cmd check` 和 `make.cmd verify-production`。
